[윤정민] [오후 9:15] 파일 가계도 시스템

제가 이 파일 가계도 시스템을 아이디어로 제안한 이유는 무엇이냐면, 반디집이나 어떤 응용 프로그램을 실행할 때, C:\Users\사용자 이름\AppData\Local\Temp 디렉토리에 각기 다른 확장자와 이름의 파일이 생성된다는 점에서였습니다. 저는 컴퓨터를 종료하기 전에 이 디렉토리에 접근해서 모든 파일을 삭제하는 루틴을 가지고 있습니다. 그리고 악성 프로그램을 실행 하면 악성 코드(서버 파일)이 이 디렉토리에 생성되는 것마저 몹시 궁금했었던 적이 있습니다. 이 프로젝트는 어느 응용프로그램을 실행하면 생성되는 파일을 실시간으로 추적하는 기능은 아니지만, 물론 다수의 컴퓨터 사용자들이 이것을 궁금해 하지 않을 것이라는 생각도 있지만, 제 궁금증에서 파생된 것이 바로 이 아이디어임을 말씀 드리고 싶습니다.

목적: 본 프로젝트는 기존 파일 시스템이 기록하지 않는 파일 간의 생성 및 파생 관계, 즉 "혈연 관계"를 자동으로 추적하고 시각화하는 시스템을 개발하는 것을 목표로 한다. 사용자의 컴퓨터 백그라운드에서 동작하는 응용 프로그램 수준의 모니터링 에이전트가 파일의 복사, 편집 후 다른 이름으로 저장 등 주요 파일 생성 이벤트를 감지하고, 이 이벤트와 현재 실행 중인 프로세스 목록을 실시간으로 교차 분석하여 파일들 사이에 부모-자식 관계의 그래프를 자동으로 구축하고, 사용자가 원할 때 언제든지 특정 파일의 기원과 전체 파생 역사를 눈으로 볼 수 있는 가계도 맵을 제공하는 시스템이다.

데이터 관리의 패러다임 전환 - 파일 이름이나 수정 날짜 같은 단편적인 정보에 의전하는 현재의 파일 관리 방식에서 벗어나, 파일 간의 "숨겨진 맥락과 역사"를 기반으로 데이터를 관리하는 새로운 패러다임을 제시한다.

작업 흐름의 자동 문서화: 데이터 분석, 디자인, 문서 작업 등에서 생성되는 수많은 중간 파일과 결과물 간의 관계를 자동으로 기록하여 작업의 재현성을 높이고 실수로 인한 데이터 유실 위험을 줄인다.

직관적인 사용자 경험 제공: 복잡한 명령어 없이, Windows 파일 탐색기 우클릭 > 파일 가계도 보기 기능을 적용해서 익숙한 인터페이스를 통해 파일의 역사 추적 및 원본을 간단하게 찾을 수 있다.

방법: 

본 시스템은 세 가지 핵심 모듈을 Python과 같은 고수준 언어 중심으로 개발하여 유기적으로 연동하는 방식으로 수행된다.

1. 백그라운드 이벤트 모니터: 사용자의 파일 시스템에서 파일 생성, 이름 변경 등 주요 이벤트를 실시간으로 감지하는 경량 에이전트로, Python의 watchdog 라이브러리를 사용하여 특정 폴더(Desktop, My Document)나 드라이브 전체의 파일 시스템 이벤트(파일 생성, 파일 수정, 파일 이름 변경)를 감지하면 알람으로 알려준다.

2. 인과관계 추론 엔진: 이벤트 발생 시점의 시스탬 상태를 분석하여 파일 간의 '부모-자식' 관계를 추론하는 핵심 요소로, Python의 psutil 라이브러리를 활용하여 이벤트가 감지된 바로 그 순간에 어떤 프로세스가 해당 파일에 접근하고 있었는지 교차 분석하여 관계를 확립한다.  

3. 데이터베이스 및 시각화 UI: 추론된 파일 관계를 SQLite 데이터베이스에 저장하고, 사용자의 요청이 있을 때 Electron 기반의 데스크톱 앱을 통해 인터랙티브한 그래프로 시각화한다. 이를 백엔드(Python,Flask)와 프론트엔드(Electron/React)가 API 통신을 통해 데이터를 주고받는 구조로 개발한다.

졸업작품 혁신성:

기술적 창의성: 응용 프로그램 수준에서의 저수준 문제 해결
전통적으로 파일 시스템의 모든 I/O를 감시하는 것은 복잡한 커널 드라이버 개발이 필요한 저수준 프로그래밍의 영역이였고, 이는 높은 진입 장벽과 개발 난이도를 가진다.
본 프로젝트는 이 문제를 두 개의 고수준 라이브러리인 watchdog와 psutil을 조합하여 해결한다.
복잡한 저수준 문제를 각기 다른 역할을하는 고수준 도구들의 시너지를 통해 해결하는 문제 해결 능력을 보여준다.

사용자는 더 이상 복잡한 로그를 해석할 필요 없이, 자신의 파일이 어떻게 태어나고 진화했는지에 대한 과정을 시각적으로 탐색할 수 있고, 전문적인 기술을 모든 사용자에게 제공함으로써 더 스마트하게 컴퓨터를 사용할 수 있도록 장려한다.

핵심 언어: Python(변동 가능성 있음, C/C++ 고려중, 언어 변경 시 후술한 API나 라이브러리도 그에 맞게 변동)
백그라운드 에이전트(모니터링 & 추론):
    - watching library: 파일 시스템 이벤트를 감지하는 방법을 학습한다.
    - psutil library: 실행 중인 프로세스 목록과 각 프로세스가 열고 있는 파일 핸들을 조회하는 방법을 학습한다.

데스크톱 UI: Electron(TypeScript + React),D3.js (시각화)
백엔드 - 프론트엔드 통신: Flask나 FastAPI를 사용하여 간단한 로컬 API 서버를 구축하고, Electron UI와 통신하는 방법을 학습한다.

온리패스는 외모 기반과 고급 인증(고급 인증이 무엇인지 잘 모르겠음)을 기반으로 사용자 간 매칭을 지원한다는 점에서

마찬가지로 우리 서비스와 비교대상이 아닌 것 같고,
Matchful(Meetch)은 소시오니크 이론 기반으로 성격 테스트를 한 후 성격 유형을 분석해서 잘 맞는 사용자들과 매칭을 지원해주는 플랫폼이나
자가 보고 기반의 검사로써 실제의 성격에 맞지 않는 방향으로 검사를 진행할 우려가 있는 데다 실제 대화 스타일이나 감정 변화를 아예 반영하지 않는 단점이 있고,
shapr는 관심사, 경력, 목표 기반으로 자기소개를 기반으로 매칭되는 서비스로 비즈니스 분야에서 유용하게 쓰이지만 마찬가지로 대화나 게시글 기반으로 성격을
파악해서 매칭을 시켜주는 서비스가 아닌 점에서 차이점이 있습니다.
즉, Matchful(Meetch)와 shapr는 서비스에서 제공하는 항목을 사용자가 지정해서 스스로를 정의하고 그것을 기반으로 매칭하는 서비스이나,
텍스트 기반으로 성격을 분석하고, 성격을 기반으로 사용자나 사용자, 조직 간 매칭을 지원하는 서비스는 존재하지 않기에 우리가 제안하는 서비스를 개발하는 것은 타당합니다.

        C_WaitResult : 매칭 결과 대기

        C_WaitResult --> C_ShowResult : 매칭 결과 수신
        C_ShowResult : 매칭 결과 표시(후보 리스트/프로필)

        C_ShowResult --> C_Idle : 종료 또는 다시 매칭
    }

    %% =====================
    %% Server + DB 영역
    %% =====================
    state "Server+DB" as ServerDB {
     


     
        S_ReceiveFile : txt 파일 수신

        S_ReceiveFile --> S_SaveRaw
        S_SaveRaw : 원본 txt를 스토리지/DB에 저장

        S_Parse : 카카오톡 포맷 파싱/(ChatMessage로 분해)

        S_SendResult : 매칭 결과(후보 리스트) 응답 생성

    }

[이승환] [오후 3:54] def score_pair(u, v, mode="friend"):
    # 게이트: 독성 차단
    if max(u.tox, v.tox) > 0.15:
        return 0.0

    S_BF = bigfive_score(u.B, v.B, mode)          # 유사/보완 혼합
    S_tone = cosine(u.S, v.S)
    P_neg = max(0, u.S[1]-0.35) + max(0, v.S[1]-0.35)
    S_tone = max(0.0, S_tone - 0.5 * P_neg)

    S_tox = 1 - ((u.tox + v.tox) / (2 * 0.15))
    S_tox = max(0.0, min(1.0, S_tox))             # 옵션: 완화에만 사용

    S_style = style_score(u.L, v.L)               # d* 보완형
    S_topic = 1 - abs((u.topic_div - v.topic_div) - 0.2)

    if mode == "friend":
        alpha, beta, gamma, delta = 0.55, 0.15, 0.20, 0.10
    else:
        alpha, beta, gamma, delta = 0.45, 0.22, 0.23, 0.10

    S = alpha*S_BF + beta*S_tone + gamma*S_style + delta*S_topic
    return max(0.0, min(1.0, S))

1. 칸반 차트 활용
우리 팀은 문제 분석부터 아이디어 구체화, 프로토타입 정의까지의 전 과정을 시각적으로 관리하기 위해 칸반 차트를 사용하였다.
칸반 보드는 “Backlog – To Do – In Progress – Review – Done” 칼럼으로 구성하고, 각 칼럼에는 사용자 조사, 시나리오 정리, 요구사항 도출, PRD 작성, 화면 설계, AI 기능 정의 등의 작업 카드를 생성하여 추적하였다.

특히 “시각장애인 쇼핑 페인포인트 조사”, “기존 바코드/OCR 한계 분석”, “초거대 AI 멀티모달 기능 정의” 같은 항목을 개별 태스크로 분리해 우선순위를 정하고, 담당자를 지정함으로써 작업 병목을 줄였다.
이렇게 칸반을 사용한 덕분에 팀원들이 현재 어떤 일이 어디까지 진행되었는지 한눈에 파악할 수 있었고, 이슈가 생긴 작업은 즉시 칼럼 이동 및 코멘트를 통해 조정할 수 있었다.

2. PRD(제품 요구사항 문서) 작성
시각장애인의 “독립적 쇼핑”이라는 목표를 구체적인 제품 수준으로 떨어뜨리기 위해 PRD를 중심 도구로 사용하였다.
먼저 문제 재정의에서 정리한 내용(맥락 정보 부재, 단순 OCR/바코드의 한계, 1+1 행사/상품 위치 인지 어려움 등)을 바탕으로, PRD 상단에 “문제 배경”과 “타깃 사용자(시각장애인, 저시력자)”를 명시했다.

그 다음 PRD에 핵심 기능을 다음과 같이 구조화했다.

핵심 가치 제안

“시각장애인이 타인 도움 없이 대형마트·편의점·쇼핑몰에서 상품 상태, 행사 정보, 위치 정보를 음성/촉각 피드백으로 실시간 파악하게 한다.”

주요 기능 요구사항

상품 인식: 스마트폰 카메라 또는 스마트 안경으로 상품 선반을 비추면 초거대 AI가 이미지+텍스트를 분석해 상품 이름, 카테고리, 기본 정보(브랜드, 용량 등)를 음성으로 안내.

맥락 정보 제공: “이 과일이 신선한지”, “유리병 콜라와 술 구분”, “1+1 행사 상품이 현재 시점에서 어떤 것인지” 등을 멀티모달·추론 기능으로 설명.

위치 안내: “오른쪽 두 번째 선반, 가슴 높이쯤에 1+1 콜라 세트가 있습니다”처럼 공간적 설명을 제공하여 사용자가 실제로 손으로 닿을 수 있게 유도.

인터랙션 방식: 음성 명령(“행사 상품만 알려줘”, “유리병 아닌 콜라 찾아줘”), 이어폰/골전도 헤드셋 출력, 진동 피드백(거리·위치 강조).

비기능 요구사항

응답 속도(실시간에 가깝게), 프라이버시(영상/음성 데이터 최소 저장), 오프라인/저속 네트워크 환경 시 폴백 전략 등.

또한 PRD에 “초거대 AI API 연동 방식(비전+텍스트 멀티모달 API, 자연어 응답 생성 API)”, “사용자 여정 시나리오(입장 → 상품 탐색 → 비교 → 결제까지)”를 서술하여, 개발·디자인·사업성 분석이 같은 문서를 기준으로 움직일 수 있도록 했다.

3. 애자일 소프트웨어 개발(Agile) 적용
개발 방식은 애자일을 따르되, 학교 프로젝트/스타트업 초기 상황에 맞게 가볍게 적용하였다.
우선 전체 프로젝트를 1~2주 단위의 스프린트로 나누고, 각 스프린트마다 “이번에 반드시 검증할 가설/기능”을 하나씩 정했다.

예시)

스프린트 1: “시각장애인 쇼핑 시나리오와 핵심 페인포인트를 구체화할 수 있는가?” → 결과물: 문제 재정의 문서, 사용자 여정 맵 초안.

스프린트 2: “초거대 AI 비전+언어 API로 상품 인식 및 간단 설명까지는 가능한가?” → 결과물: 간단한 데모 플로우 설계/모형.

스프린트 3: “행사 정보·위치 정보 등 맥락 설명을 어떻게 말해줘야 실제로 도움이 되는가?” → 결과물: 대화 플로우, 음성 인터랙션 시나리오.

각 스프린트 시작 시에는 칸반 보드를 기준으로 계획 미팅을 진행해 “이번 스프린트 To Do”를 확정했고, 스프린트 종료 시에는 간단한 리뷰를 통해

무엇이 잘 되었는지(예: 맥락 설명 문구가 시각장애인 피드백에 긍정적),

무엇을 개선해야 하는지(예: 응답이 길어서 피로감 증가, UI 피드백 부족)
를 정리하여 다음 스프린트 계획에 반영했다.

사고력 강화 포인트:

처음에는 떠올리지 못한 “공간 안내, 건강 정보, 행사 정보”까지 문제 범위를 확장하면서 발산적 사고 능력을 키움.

마인드맵(Mind Map) 적용 예제

적용 방법:

중앙에 “시각장애인 쇼핑 문제”를 적고, 가지를 “정보 접근”, “이동/위치”, “상품 비교”, “감정/심리” 등으로 뻗어나가며 연관 이슈를 연결함.

도출 예시:

“정보 접근” → “성분표 인식” → “알레르기 알림”, “저당 제품 추천”

“이동/위치” → “선반 위치 안내” → “오른쪽 두 칸 옆, 허리 높이” 같은 구체적 언어 설계

사고력 강화 포인트:

문제를 단순히 “안 보인다”가 아니라, 여러 하위 요소(정보, 이동, 비교, 감정)로 구조화하면서 분석적 사고를 훈련.

5 Whys(5번 왜? 기법) 적용 예제

적용 방법:

“왜 시각장애인은 독립적으로 쇼핑하기 어려운가?”를 시작으로 최소 5번 “왜?”를 반복 질문.

예시 전개:

Q1: 왜 독립 쇼핑이 어려운가?
→ 필요한 정보를 스스로 얻기 어렵다.

Q2: 왜 정보를 얻기 어려운가?
→ 글자, 가격표, 행사 안내를 눈으로 볼 수 없다.

Q3: 왜 기존 기술로도 해결이 안 되나?
→ OCR/바코드 리더는 ‘이름·가격’ 정도만 읽어주고, 신선도·위험도·행사 위치 같은 맥락은 못 준다.

Q4: 왜 맥락이 중요한가?
→ 실제 선택과 구매 결정은 “이게 나에게 적합한가?”라는 맥락 판단이 핵심이기 때문.

Q5: 그래서 우리는 무엇을 해야 하나?
→ 멀티모달 초거대 AI를 이용해 “맥락·추론 기반 설명”을 해주는 서비스를 설계해야 한다.

사고력 강화 포인트:

표면적인 문제(‘안 보임’)에서 한 단계씩 깊게 내려가 “정말 해결해야 할 코어 문제(맥락 기반 의사결정)”를 찾는 논리적 사고를 강화.

SCAMPER 기법 적용 예제
(기존 바코드/OCR 방식에 SCAMPER를 적용한 예시)

Substitute(대체):

단순 바코드 인식을 “이미지+텍스트+위치 정보를 한 번에 처리하는 초거대 AI”로 대체.

Combine(결합):

상품 인식 + 매장 지도 + 사용자 건강 프로필을 결합해 “맞춤형 추천 + 위치 안내”를 동시에 제공.

Adapt(응용):

비장애인용 AR 쇼핑 기술(카메라로 비추면 정보 뜨는 것)을 시각장애인에게 맞게 “음성/진동 피드백” 중심으로 변형.

Modify(수정):

기존 OCR 앱의 “긴 텍스트를 그대로 읽어주는 방식”을 “핵심 정보만 요약해 주는 방식”으로 수정.

사고력 강화 포인트:

이미 있는 솔루션(바코드/OCR, AR 쇼핑)을 비판적으로 뜯어보고, 어떤 부분을 바꾸고, 합치고, 응용할 수 있을지 체계적으로 생각하게 됨.
[이승환] [오후 6:24] 5. TU-PBL 문제 해결 방안 도출


시각장애인이 스마트폰(또는 스마트 안경) 카메라로 선반·상품을 비추면, 초거대 AI가 이미지+텍스트를 분석해 상품 정보, 신선도, 행사 여부, 위치 정보를 음성으로 설명해 주는 서비스.

“단순 바코드/텍스트 인식”이 아니라, “맥락과 추론”을 포함해 비장애인과 유사한 수준의 판단 정보를 제공하는 것을 목표로 함.

2. 구체 기능으로 본 해결 방안
상품·행사 인식 기능

카메라에 비친 상품들을 인식하고, 이름·카테고리·가격·행사 여부(1+1, 할인 등)를 한 번에 요약해서 안내.

맥락 정보 제공 기능

과일의 신선도(색, 상태), 유리병 콜라와 술 구분, 알코올/카페인 여부 등 “단순 라벨 이상”의 정보를 설명.

위치·방향 안내 기능

“오른쪽 두 칸 옆, 가슴 높이 선반”처럼 공간 정보를 말로 안내해 사용자가 실제로 손으로 찾을 수 있게 도움.

개인 맞춤 추천 기능

알레르기·질환·선호(저당, 저카페인, 예산 등)를 프로필로 저장하고, 그 기준에 맞는 상품을 추천·비교해 줌.

이 전체를 초거대 AI의 멀티모달 API(비전+언어)와 자연어 대화 기능을 결합해 구현하는 것이 핵심 해결 방안이다.

3. 왜 이 해결 방안을 선택했는지 – 다양한 관점
사용자 관점(시각장애인 입장)

실제 페인포인트와 직접 연결됨:

현재 시각장애인은 “무엇인지”뿐 아니라 “좋은지/안전한지/어디 있는지”를 알기 어렵다는 문제를 겪는다.

이 방안은 상품 정보, 맥락(신선도·위험도), 위치, 비교·추천까지 한 번에 다루기 때문에 사용자가 느끼는 핵심 불편을 직접 겨냥한다.

“완전한 소비 주권”에 가까움:

타인의 주관적 도움 없이 스스로 선택·비교·결정할 수 있도록 설계되어, 단순 보조가 아니라 “독립적 쇼핑”에 더 가깝다.

기술 관점(초거대 AI 활용성)

초거대 AI의 강점을 그대로 활용:

멀티모달 처리(이미지+텍스트), 자연어 이해·생성, 추론 능력은 지금 상용화된 초거대 AI의 대표적인 장점이다.

이 해결 방안은 별도의 복잡한 규칙·모델을 새로 만드는 것보다, 이미 검증된 초거대 AI API를 활용해 빠르게 구현·검증할 수 있다.

확장성·업데이트 용이:

모델이 계속 업데이트되면, 자동으로 더 좋은 인식·설명을 제공할 수 있어 장기적으로 유지·개선이 쉽다.

사업성·비즈니스 관점

시장 니즈 명확:

시각장애인 쇼핑 보조 솔루션은 이미 여러 스타트업과 연구에서 시도 중일 정도로 수요가 검증된 영역이고, 오프라인·온라인 유통사와의 제휴 가능성이 크다.

다양한 수익 모델 가능:

B2C: 시각장애인 대상 구독형 프리미엄 기능(맞춤 추천, 건강 관리 연동 등).

B2B: 대형마트·편의점·쇼핑몰에 “접근성 향상 솔루션”으로 납품하거나, 행사·진열 최적화 데이터 제공.

API/SDK 제공: 다른 보조공학 기기(스마트 지팡이, 스마트 글라스 등)와 연동 가능한 플랫폼으로 확장 가능.

구현 현실성·운영 관점

하드웨어 부담이 적음:

사용자의 스마트폰과 기존 매장 환경을 그대로 활용할 수 있어, 고가의 전용 장비나 매장 전체 리모델링이 필요하지 않다.

단계적 도입이 가능:

1단계: 상품 인식+기본 정보 음성 안내

2단계: 행사/위치 안내, 간단 추천

3단계: 개인 맞춤 추천, 매장 동선 내비게이션

이런 식으로 MVP→고도화 순서로 점진적 개발·출시가 가능해 리스크를 분산할 수 있다.

[윤정민] [오후 10:03] 1. Google Cloud Platform(GCP) 프로젝트 생성
가장 먼저 구글 클라우드 콘솔에 접속하여 프로젝트를 만들어야 하고
    console.cloud.google.com에 접속하여 구글 아이디로 로그인
    상단 메뉴에서 프로젝트 만들기(Create Project)를 클릭하여 새 프로젝트를 생성 (프로젝트 이름은 자유롭게 설정)
    생성된 프로젝트 ID 복사

2. API 액세스 권한 요청 (필수)
Perspective API는 프로젝트를 만들었다고 바로 쓸 수 있는 것이 아니라, 별도의 승인 요청을 해야 활성화됨
    Perspective API 액세스 요청 폼으로 이동
    폼에 앞서 복사한 GCP 프로젝트 ID와 사용 목적 등을 입력하고 제출
    보통 1시간 이내(빠르면 몇 분 내)에 승인 이메일이 도착함

3. API 활성화 및 키 발급
승인 메일을 받으면 다시 구글 클라우드 콘솔로 돌아가서
    콘솔의 'API 및 서비스' > '라이브러리' 메뉴로 이동
    'Perspective Comment Analyzer API'를 검색하고 사용(Enable) 버튼 클릭
   (승인 전에는 검색되지 않거나 활성화되지 않을 수 있음)
    API 및 서비스 > 사용자 인증 정보(Credentials)탭으로 이동
    사용자 인증 정보 만들기를 누르고 API 키를 선택하면 키가 발급됨
